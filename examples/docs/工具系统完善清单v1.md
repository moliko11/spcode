## P0：先补元信息，不然后面很多策略没法挂

### 1. 扩展 `ToolSpec`

你现在的 `ToolSpec` 只有 `allowed_roles / require_approval / timeout_s / readonly`，够跑，但不够做策略。建议补这些字段：

```
risk_level: Literal["low", "medium", "high", "critical"] = "low"
side_effect: Literal["none", "local_fs", "shell", "network", "external_api", "vcs"] = "none"
sandbox_required: bool = False
network_required: bool = False
writes_workspace: bool = False
cache_policy: Literal["success_only", "all", "none"] = "success_only"
max_retries: int | None = None
approval_policy: Literal["never", "always", "on_write", "on_high_risk"] = "never"
```

作用：

- &#x20;决定是否自动执行&#x20;
- &#x20;决定是否必须沙箱&#x20;
- &#x20;决定失败要不要缓存&#x20;
- &#x20;决定默认重试次数&#x20;

### 2. 给现有工具补分类

先把你现有工具标好：

- `calculator`：`low / none / no sandbox / no approval`&#x20;
- `get_current_time`：`low / none`&#x20;
- `read_file`：`medium / local_fs / sandbox_required=True / writes_workspace=False`&#x20;
- `write_note`：`high / local_fs / sandbox_required=True / writes_workspace=True / approval_policy=always`&#x20;

## P0：先把 `ToolExecutor` 变成真正的执行链

你现在执行链已经有雏形了，下一步收敛成固定顺序：

### 4. 固定执行顺序

建议在 `execute()` 里强制按这个顺序：

```
load spec
-> validate args
-> permission check
-> approval check
-> sandbox policy resolve
-> idempotency lookup
-> emit TOOL_STARTED
-> execute with timeout/retry
-> normalize result
-> cache result
-> emit TOOL_FINISHED/FAILED
```

### 5. 调整幂等缓存策略

你现在会缓存所有结果，建议改成按 `cache_policy`：

- `success_only`：只缓存成功&#x20;
- `all`：成功失败都缓存&#x20;
- `none`：不缓存&#x20;

建议默认：

- `calculator / read_file`：`success_only`&#x20;
- `write_note / shell`：谨慎，默认 `success_only`&#x20;
- &#x20;网络抖动类失败不要缓存&#x20;

### 6. 重试次数按工具覆盖

现在是全局 `RetryPolicy`。建议：

- &#x20;全局默认还保留&#x20;
- `ToolSpec.max_retries` 可覆盖全局&#x20;
- &#x20;只对“可重试异常”生效&#x20;

建议异常分层：

- &#x20;不重试：`GuardrailViolation / PermissionDenied / HumanInterventionRequired / ValueError`&#x20;
- &#x20;可重试：`TimeoutError / ConnectionError / 临时 5xx / shell 临时资源错误`&#x20;

## P0：审批系统从布尔开关升级成策略

### 7. `ApprovalController` 支持策略化

现在 `require_approval` 太粗。改成：

```
def needs_approval(spec, call) -> bool:
    if spec.approval_policy == "never": ...
    if spec.approval_policy == "always": ...
    if spec.approval_policy == "on_write": ...
    if spec.approval_policy == "on_high_risk": ...
```

### 8. 支持 `edit_arguments`

你已经在 `resume()` 里留了 `edited_arguments`，很好。下一步补两个点：

- &#x20;审批通过后重新走 `validate_tool_args`&#x20;
- &#x20;审批记录写进 `ToolResult.approved_by` 或 `state.metadata`&#x20;

### 9. 审批对象落地字段

建议 `HumanInterventionRequest.context` 里固定包括：

```
{
  "tool_name": ...,
  "arguments": ...,
  "risk_level": ...,
  "side_effect": ...,
  "reason": ...,
}
```

前端和日志都更好用。

## P0：bash / shell 单独成体系，不要当普通工具

### 10. 新建 `ShellToolSpec`

shell 不能只用普通 `ToolSpec`，建议单独规范：

```
allowed_commands: list[str] = []
blocked_patterns: list[str] = []
working_dir_mode: Literal["workspace_only"] = "workspace_only"
env_policy: Literal["empty", "safe_default", "inherit_filtered"] = "safe_default"
shell_mode: Literal["bash", "sh", "powershell"] = "bash"
capture_output_limit: int = 20000
```

### 11. 新建 `ShellExecutor`

单独处理：

- &#x20;命令拼装&#x20;
- &#x20;工作目录限制&#x20;
- &#x20;环境变量过滤&#x20;
- &#x20;stdout/stderr 捕获&#x20;
- &#x20;exit\_code&#x20;
- &#x20;timeout&#x20;
- &#x20;denylist&#x20;

### 12. 最小沙箱要求

bash/shell v1 先做到：

- &#x20;只能在 `WORKSPACE_DIR` 下运行&#x20;
- &#x20;默认不继承全量环境变量&#x20;
- &#x20;默认禁止危险命令模式&#x20;
- &#x20;默认需要审批&#x20;
- &#x20;输出截断上限&#x20;
- &#x20;超时强杀&#x20;

哪怕 v1 还没上 Docker，也要先做这个“轻沙箱”。
